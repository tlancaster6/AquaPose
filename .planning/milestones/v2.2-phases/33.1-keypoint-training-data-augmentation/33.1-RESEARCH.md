## User Constraints (from CONTEXT.md)

### Locked Decisions

**Augmentation types:**
- Geometric transforms (full set): horizontal flip, random rotation (±15°), random scale (0.8–1.2×), random translate (±10%)
- Color/photometric transforms (standard): brightness, contrast, slight hue/saturation jitter
- Crop jitter: random crop offset (±5–10% of bbox) and scale jitter to simulate imperfect detection boxes at inference time
- Horizontal flip logic: mirror x-coordinates only, no keypoint index swapping — all 6 keypoints (Nose, Head, Spine1, Spine2, Spine3, Tailbase) are midline, not bilateral

**Keypoint handling during augmentation:**
- Out-of-bounds keypoints: set coordinates to 0.0 and visibility to 0 (mark invisible), matching the existing COCO visibility=0 convention
- Minimum visible keypoints: discard augmented samples with fewer than 3 of 6 visible keypoints (matches Phase 33's `min_observed_keypoints` inference contract)
- Visibility mask returned by dataset: `KeypointDataset.__getitem__` returns `(image, keypoints, visibility_mask)` — visibility derived from COCO `v` flags and updated by augmentation transforms when keypoints go out-of-bounds
- Masked loss: update `train_pose` MSE loss to only compute on visible keypoints (visibility > 0) — prevents learning to predict (0,0) for missing points

**Pipeline integration:**
- Architecture: dataset-level composable transform passed to `KeypointDataset` — training loop unchanged (except loss and unpack)
- Scope: keypoint-specific augmentation, not shared with U-Net segmentation training
- Always enabled: no CLI toggle — augmentation is always active during training
- Hardcoded defaults: transform parameters are sensible defaults baked into the code, not configurable

**Training regime:**
- Online augmentation: random transforms applied on-the-fly each epoch (not pre-generated)
- Validation set: un-augmented only — clean images for measuring real-world performance
- Augmentation ratio: 1 augmented copy + 1 clean copy per sample per epoch (effective 2× dataset size, 156 samples per epoch from 78 originals)

## Summary

Phase 33.1 adds keypoint-aware data augmentation to the existing `KeypointDataset` and `train_pose` pipeline in `src/aquapose/training/pose.py`. The dataset is small (78 samples, mostly 6/6 visible keypoints, 13 with partial visibility), making augmentation critical for generalization. All required transforms are available in the installed `torchvision 0.25.0` via `torchvision.transforms.v2` with native `tv_tensors.KeyPoints` support — keypoints are propagated correctly through geometric transforms with no custom math required.

The work spans three interconnected changes: (1) augment the `KeypointDataset.__getitem__` return to a 3-tuple `(image, keypoints, visibility_mask)`, (2) wire up `torchvision.transforms.v2` transforms that jointly transform image and `tv_tensors.KeyPoints`, and (3) update `train_pose` to use masked MSE loss and a `ConcatDataset([clean, augmented])` training set. The val set stays augmentation-free.

Key implementation insight: augmentation must happen in 128×128 pixel coordinate space (after resize from native crop size), with keypoints mapped to pixel coords before transforms, then normalized to [0,1] after. The `REQUIREMENTS.md` explicitly mandates `torchvision.transforms.v2` over albumentations.

**Primary recommendation:** Use `torchvision.transforms.v2` with `tv_tensors.KeyPoints`, `RandomHorizontalFlip` + `RandomAffine` + `ColorJitter`. `RandomAffine(degrees=15, translate=(0.1,0.1), scale=(0.8,1.2))` covers geometric + crop jitter in one transform. No additional dependencies needed.

### Pattern 1: tv_tensors.KeyPoints Augmentation

**What:** Wrap keypoints as `tv_tensors.KeyPoints` so that `torchvision.transforms.v2` applies the same geometric transform to the image and the keypoints simultaneously.

**When to use:** Any time geometric transforms (flip, rotate, affine) need to be applied consistently to both image and coordinate annotations.

**Example (verified in hatch env with torchvision 0.25.0):**

```python
from torchvision import tv_tensors
from torchvision.transforms import v2
import torch

# Build transform pipeline (hardcoded defaults)
_AUGMENT_TRANSFORM = v2.Compose([
    v2.RandomHorizontalFlip(p=0.5),
    v2.RandomAffine(degrees=15, translate=(0.1, 0.1), scale=(0.8, 1.2)),
    v2.ColorJitter(brightness=0.3, contrast=0.3, saturation=0.3, hue=0.05),
])

# In KeypointDataset.__getitem__, after resize to 128x128:
img_tv = tv_tensors.Image(image_tensor_uint8)   # (3, 128, 128) uint8
kps_pixel = tv_tensors.KeyPoints(
    kp_pixel_coords,    # (6, 2) float32, in [0, 128] pixel space
    canvas_size=(128, 128)
)

img_aug, kps_aug = _AUGMENT_TRANSFORM(img_tv, kps_pixel)

# Detect out-of-bounds keypoints and update visibility
x = kps_aug[:, 0]
y = kps_aug[:, 1]
h, w = kps_aug.canvas_size
oob_mask = (x < 0) | (x >= w) | (y < 0) | (y >= h)
visibility[oob_mask] = False

# Zero out OOB keypoint coordinates (COCO convention)
kp_pixel_coords_final = kps_aug.clone()
kp_pixel_coords_final[oob_mask] = 0.0
```

### Pattern 2: Masked MSE Loss

**What:** Compute MSE only on keypoints where `visibility > 0`. Prevents the model from learning to predict (0,0) for invisible keypoints.

**When to use:** Any keypoint regression with partial visibility annotations or augmentation that invalidates some keypoints.

**Example (verified):**

```python
def _masked_mse_loss(
    pred: torch.Tensor,      # (B, n_keypoints * 2)
    target: torch.Tensor,    # (B, n_keypoints * 2)
    visibility: torch.Tensor,  # (B, n_keypoints) bool or float
) -> torch.Tensor:
    vis_expanded = visibility.float().repeat_interleave(2, dim=1)  # (B, n_keypoints*2)
    diff_sq = (pred - target) ** 2
    masked = diff_sq * vis_expanded
    n_visible = vis_expanded.sum().clamp(min=1.0)
    return masked.sum() / n_visible
```

The division by `clamp(min=1.0)` ensures no division by zero when all keypoints are invisible (returns 0.0 loss, which is correct — nothing to learn from).

### Pattern 3: Clean + Augmented ConcatDataset

**What:** Create two `KeypointDataset` instances (clean and augmented) over the same training indices, then combine with `ConcatDataset` for 2× effective epoch size.

**When to use:** Online augmentation with 1:1 clean-to-augmented ratio per epoch.

**Example:**

```python
from torch.utils.data import ConcatDataset, Subset

# Both datasets read same files; augmented one applies transforms
train_clean = KeypointDataset(json_path, image_root, augment=False)
train_aug   = KeypointDataset(json_path, image_root, augment=True)

# Split indices from clean dataset (stratified_split works on _images protocol)
train_indices, val_indices = stratified_split(train_clean, val_fraction=val_split)

# Training: 1 clean + 1 augmented per sample = 2x epoch size
train_dataset = ConcatDataset([
    Subset(train_clean, train_indices),
    Subset(train_aug, train_indices),
])

# Validation: clean only
val_dataset = Subset(train_clean, val_indices)
```

### Pattern 4: KeypointDataset 3-Tuple Return

**What:** Extend `KeypointDataset.__getitem__` to return `(image, keypoints, visibility_mask)`.

**Critical:** The `visibility_mask` must be derived from COCO `v` flags initially, then updated by augmentation (OOB keypoints set to invisible). The clean (non-augmented) path still returns a visibility mask derived purely from COCO annotations.

**Coordinate space protocol:**
1. Load native-size image and resize to 128×128
2. Map keypoints to 128×128 pixel space: `x_px = x_coco / orig_w * 128`
3. Apply augmentation transforms to `tv_tensors.Image` + `tv_tensors.KeyPoints`
4. Detect OOB, update visibility
5. Normalize to [0,1]: `x_norm = x_px / 128`
6. Zero out coordinates for invisible keypoints: `kp_flat[~vis] = 0.0`

### Anti-Patterns to Avoid

- **Normalizing before augmentation:** Applying transforms to [0,1]-normalized coordinates breaks `tv_tensors.KeyPoints` which expects pixel-space coordinates matching `canvas_size`. Always augment in pixel space, normalize after.
- **Augmenting the val set:** Validation must use un-augmented images only. The `augment=False` path is used for val dataset construction.
- **Modifying visibility for the clean path:** For clean samples, visibility comes only from COCO `v` flags. Do not set any keypoints invisible based on geometric transforms that were never applied.
- **Missing `clamp(min=1)` in masked loss:** Without it, batches where all keypoints happen to be invisible produce NaN loss, which propagates to parameter gradients.
- **Forgetting to update `_mean_keypoint_error`:** The validation error metric must also be masked — otherwise the 13 partial-visibility annotations contaminate val metrics with the error at (0,0).

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Apply same rotation to image + keypoints | Custom rotation + trig math | `torchvision.transforms.v2.RandomAffine` with `tv_tensors.KeyPoints` | v2 handles affine coordinate math exactly, including border fill |
| Horizontal flip + keypoint mirror | Custom flip + `x = W - x` | `v2.RandomHorizontalFlip` with `tv_tensors.KeyPoints` | v2 handles it automatically |
| Crop jitter (translate + scale simulation) | Separate `RandomResizedCrop` | Subsume into `RandomAffine(translate=..., scale=...)` | One transform covers all geometric augmentations |
| Out-of-bounds detection | Canvas boundary checks | Post-transform OOB check (`x < 0 | x >= W | y < 0 | y >= H`) | Simple, correct — `SanitizeKeyPoints` REMOVES keypoints (wrong behavior); `ClampKeyPoints` clips them (also wrong). Manual check then zero/mask is right. |

**Key insight:** `torchvision.transforms.v2.SanitizeKeyPoints` and `ClampKeyPoints` are NOT the right tool here. `SanitizeKeyPoints` removes entire keypoints from the tensor (changes shape — breaks fixed-N regression). `ClampKeyPoints` clips to edge (incorrect — an occluded tail should not be predicted at the edge). The correct approach is a post-transform OOB mask that zeros coordinates and sets visibility to 0, preserving tensor shape.

## Common Pitfalls

### Pitfall 1: torchvision v2 Transforms Require `tv_tensors` Wrapped Inputs

**What goes wrong:** Passing a plain `torch.Tensor` for keypoints to a `v2` transform — the transform silently treats it as a generic tensor and does NOT apply geometric corrections. The keypoints stay in pre-transform coordinates while the image is transformed.

**Why it happens:** `v2` transforms only apply coordinate-aware processing to recognized `tv_tensors` types (`Image`, `BoundingBoxes`, `KeyPoints`, `Mask`). Plain tensors pass through unchanged.

**How to avoid:** Always wrap keypoints with `tv_tensors.KeyPoints(..., canvas_size=(H, W))` before passing to the `v2.Compose` pipeline. Verified this works correctly in torchvision 0.25.0.

**Warning signs:** Keypoint coordinates don't change after a `RandomRotation(degrees=90)` — they should change significantly.

### Pitfall 2: Coordinate Space Mismatch

**What goes wrong:** Applying `tv_tensors.KeyPoints` with normalized [0,1] coordinates when `canvas_size=(128,128)` causes incorrect OOB detection and incorrect coordinate transforms (the transforms assume pixel coordinates matching the canvas).

**Why it happens:** `tv_tensors.KeyPoints` uses `canvas_size` as the pixel bounds for OOB checking and as the reference frame for geometric transforms. Normalized coords (max ~1.0) are always "within" a 128×128 canvas and produce wrong transform outputs.

**How to avoid:** Convert to pixel space before creating `tv_tensors.KeyPoints`. The protocol is: resize image to 128×128, map `x_px = x_coco / orig_w * 128`, create `KeyPoints(..., canvas_size=(128, 128))`, augment, then normalize `x_norm = x_px_aug / 128`.

### Pitfall 3: DataLoader 3-Tuple Unpacking

**What goes wrong:** The existing `train_pose` loop unpacks `for images, targets in train_loader:` — this raises `ValueError: too many values to unpack` after `KeypointDataset` is changed to return 3-tuples.

**Why it happens:** `DataLoader` collates whatever the dataset returns; changing from 2-tuple to 3-tuple breaks all existing unpack sites.

**How to avoid:** Update the training loop to `for images, targets, visibility in train_loader:` and replace `mse_loss(pred, targets)` with `_masked_mse_loss(pred, targets, visibility)`. Also update the val loop and `_mean_keypoint_error` call.

### Pitfall 4: Retry Loop in `__getitem__` Must Be Bounded

**What goes wrong:** An infinite retry loop for `<3 visible keypoints` could deadlock or take arbitrarily long if augmentation parameters are extreme.

**Why it happens:** Geometric transforms with large rotation on narrow fish crops can push keypoints OOB.

**How to avoid:** Cap retries at 10. After 10 attempts, return the last augmented result regardless of visible keypoint count (the masked loss handles it gracefully — samples with many invisible keypoints contribute less gradient, which is correct behavior). In practice with ±15° rotation and 0.8–1.2× scale, the 78-sample dataset rarely triggers OOB because fish are centered in crops.

### Pitfall 5: `stratified_split` Called on Wrong Dataset Instance

**What goes wrong:** Calling `stratified_split` on the augmented `KeypointDataset` instance is fine (it uses `_images` protocol), but it's confusing to read. More importantly, the `train_indices` must be reused identically for both `Subset(clean, train_indices)` and `Subset(aug, train_indices)` — calling `stratified_split` twice with the same seed gives the same result, but it's a footgun.

**How to avoid:** Call `stratified_split` once on the clean dataset. Reuse the returned `train_indices` and `val_indices` for both Subset constructions.

### Pitfall 6: Actual Annotations Already Contain Keypoints with Negative Coordinates

**What goes wrong:** Some COCO annotations in the training set have keypoints with `y < 0` (verified: annotation `img_2` has y ≈ -0.9). If the clean path doesn't handle this, the visibility mask will be wrong for the baseline case.

**Why it happens:** The crops were extracted with bounding boxes that extend slightly outside the image frame. The annotation tool recorded keypoints relative to the crop, some of which land just outside the crop edge.

**How to avoid:** The visibility mask for the clean path should be: `visible = (v > 0) AND (x_px >= 0) AND (x_px < 128) AND (y_px >= 0) AND (y_px < 128)`. This is already implicitly correct since the current code only assigns coords when `v > 0`, but the explicit OOB check should be added to the clean path too for robustness.

### Masked MSE Loss in train_pose Loop

```python
# Replace: loss = torch.nn.functional.mse_loss(pred, targets)
# With:
def _masked_mse_loss(
    pred: torch.Tensor,
    target: torch.Tensor,
    visibility: torch.Tensor,
) -> torch.Tensor:
    """MSE loss computed only on visible keypoints.

    Args:
        pred: (B, n_keypoints * 2) predicted coordinates in [0, 1].
        target: (B, n_keypoints * 2) ground-truth coordinates in [0, 1].
        visibility: (B, n_keypoints) bool mask, True = visible.

    Returns:
        Scalar loss tensor.
    """
    vis_exp = visibility.float().repeat_interleave(2, dim=1)  # (B, n_kp*2)
    diff_sq = (pred - target) ** 2
    n_visible = vis_exp.sum().clamp(min=1.0)
    return (diff_sq * vis_exp).sum() / n_visible
```

### Masked Validation Error Metric

```python
def _masked_keypoint_error(
    pred: torch.Tensor,
    target: torch.Tensor,
    visibility: torch.Tensor,
) -> float:
    """Mean Euclidean distance on visible keypoints only."""
    n = pred.shape[1] // 2
    p = pred.view(-1, n, 2)
    t = target.view(-1, n, 2)
    v = visibility.float()          # (B, n)
    dist = torch.sqrt(((p - t) ** 2).sum(dim=-1))  # (B, n)
    n_visible = v.sum().clamp(min=1.0)
    return float((dist * v).sum() / n_visible)
```

## Open Questions

1. **`torchvision.transforms.v2` + uint8 vs float32 images**
   - What we know: `ColorJitter` in v2 works on both uint8 and float32 `tv_tensors.Image`. Applying it on uint8 before the `/255.0` conversion avoids floating-point accumulation of color jitter.
   - What's unclear: Whether `RandomAffine` on uint8 images introduces quantization artifacts at low scale factors.
   - Recommendation: Apply all transforms on uint8, convert to float32 at the end of `__getitem__`. This is the standard torchvision v2 pattern.

2. **Retry strategy edge case**
   - What we know: With the 78-sample dataset and ±15° rotation + 0.8–1.2× scale, very few samples will hit <3 visible after augmentation (fish are well-centered in crops, verified by looking at coordinate ranges).
   - What's unclear: Whether a retry cap of 10 is enough for all edge cases.
   - Recommendation: Cap at 10 retries, then return last result regardless. The masked loss handles sparse visibility gracefully (contributes less gradient, not wrong gradient).
