# Phase 43.1: Association Tuning - Context

**Gathered:** 2026-03-02
**Status:** Ready for planning

<domain>
## Phase Boundary

Tune cross-camera association parameters so that reconstruction receives sufficient multi-view data. The association stage is currently too aggressive — only 6-7 of 9 fish get multi-camera coverage, producing a 12-15% reconstruction rate. This phase sweeps existing AssociationConfig parameters to maximize fish yield, with minimal code changes limited to wiring (new config fields, harness extension). Algorithmic changes to scoring/clustering/refinement logic are out of scope.

Reference analysis: `.planning/inbox/association_tuning_analysis.md`

</domain>

<decisions>
## Implementation Decisions

### Tuning tooling
- Standalone script (`scripts/tune_association.py`), consistent with `tune_threshold.py` pattern
- Full auto sweep — script sweeps all parameters in priority order, auto-carries best from each stage, one command, one big output
- Runs through the eval harness (`run_evaluation()`) for each config — reuses existing Tier 1 metrics plus association-level metrics
- Skip Tier 2 (leave-one-out) during the main sweep for speed; compute full Tier 1+2 only for the top N configurations
- Console-only output (no JSON/files) — ephemeral, consistent with tune_threshold.py

### Harness wiring
- Extend `run_evaluation()` with an `association_overrides` dict parameter that patches AssociationConfig fields before running
- This is the only harness code change — no new metrics classes or schema changes

### Scope boundary
- Parameter tuning only on existing AssociationConfig fields
- Ghost penalty ablation via setting `ghost_pixel_threshold` to a very large value (existing field) — no code change to scoring logic
- Minimal wiring changes allowed: new config fields (e.g., `ghost_penalty_enabled` toggle) to make ablation cleaner
- No algorithmic changes to scoring, clustering, refinement, or must-not-link logic
- Findings about algorithmic issues (ghost penalty structure, eviction-only MNL) are recorded for a future phase but not acted on

### Sweep protocol (from analysis doc Section 6)
- Priority order: `ray_distance_threshold` → `score_min` → ghost penalty ablation → `eviction_reproj_threshold`
- Best value from each stage carries forward as the baseline for the next
- Secondary parameters (`early_k`, `leiden_resolution`) included only if primary sweep doesn't reach target

### Success criteria
- Target: 9 fish reconstructed per frame (all fish in tank)
- Yield is the primary metric — reprojection error degradation is acceptable; error will be addressed in Phase 44 (DLT threshold tuning)
- If sweep can't reach 9 fish with parameter tuning alone, present results to user for decision on whether to expand scope
- Winning config values become the new defaults in `engine/config.py` AssociationConfig, recorded in PROJECT.md Key Decisions table

### Claude's Discretion
- Exact sweep ranges for each parameter (analysis doc suggests reasonable starting ranges)
- Number of top-N configs for full Tier 2 evaluation
- Association-level metrics to print (multi-camera yield, singleton rate, cluster confidence — from analysis Section 5)
- Script CLI interface design (argparse flags, defaults)

</decisions>

<specifics>
## Specific Ideas

- The analysis doc (`.planning/inbox/association_tuning_analysis.md`) Section 5 proposes 6 metrics: reconstruction coverage rate, multi-camera yield, reprojection error, LOO stability, singleton rate, cluster consistency score. Use these as the reporting framework.
- The analysis doc Section 3 identifies `score_min` and `ray_distance_threshold` as the combined primary gatekeeper — the scoring formula `score = inlier_fraction × (1 - mean_ghost_ratio) × overlap_weight` means both interact multiplicatively.
- Ghost ablation should use a very large `ghost_pixel_threshold` (e.g., 10000) rather than adding a disable flag — this tests the effect without any code changes.

</specifics>
