## Summary

Phase 43.1 sweeps existing `AssociationConfig` parameters to maximize the number of fish that receive multi-camera coverage and can be reconstructed. The current system produces 6-7 reconstructed fish per frame out of 9 total (12-15% coverage rate). The root cause — identified in the analysis document — is that `ray_distance_threshold` (0.03m) and `score_min` (0.3) together gate out legitimate cross-camera matches whose ray distances exceed the refractive calibration error budget. Ghost penalty and eviction threshold are secondary contributors.

The implementation has two parts: (1) extend `run_evaluation()` in `evaluation/harness.py` with an `association_overrides` parameter that patches `AssociationConfig` before running, and (2) write `scripts/tune_association.py` that sweeps parameters in priority order and prints a console-only comparison report.

The key architectural insight is that the current fixture format (`midline_fixtures.npz`) stores **post-association** data — `fish_id` in the NPZ already reflects cross-camera identity assigned by the Association stage. The sweep therefore cannot re-run association from tracklet data; instead, `run_evaluation()` with `association_overrides` must be understood as patching a new association-level re-grouping step into the harness, OR the fixture must store raw per-camera tracklet data that the harness can re-associate. This design decision must be resolved in planning (see Open Questions).

**Primary recommendation:** Resolve the fixture-vs-re-association architecture question before writing plan tasks. If the fixture already embeds per-fish, per-camera data that is independent of association quality (i.e., all camera views for all tracklets were captured), then `association_overrides` can simply re-weight or re-filter the existing MidlineSet. If not, a separate pre-association fixture or a full pipeline re-run is needed.

## Architecture Patterns

### Recommended Project Structure


### Pattern 1: Config Patching via `dataclasses.replace()`

**What:** Create a modified copy of a frozen dataclass without mutation.
**When to use:** Every sweep iteration — apply `association_overrides` dict onto the baseline `AssociationConfig`.

```python
# Source: Python stdlib docs + existing codebase pattern
import dataclasses
from aquapose.engine.config import AssociationConfig

baseline = AssociationConfig()  # frozen
overrides = {"ray_distance_threshold": 0.05, "score_min": 0.25}
patched = dataclasses.replace(baseline, **overrides)
```

This is safe with `frozen=True` because `replace()` constructs a new instance.

### Pattern 2: Extended `run_evaluation()` Signature

**What:** Add `association_overrides: dict | None = None` as a keyword argument to `run_evaluation()`. The harness uses the overrides to patch `AssociationConfig` before executing association.

**Critical design question (see Open Questions):** Whether the harness re-runs association on raw tracklet data from the fixture, or whether the `association_overrides` parameter only adjusts reconstruction-time filtering.

**What the harness currently does:**
```python
# harness.py — existing flow
fixture = load_midline_fixture(fixture_path)
# fixture.frames: tuple[MidlineSet, ...] where MidlineSet = dict[fish_id, dict[cam_id, Midline2D]]
# fish_id here is post-association global identity — already grouped across cameras
```

The fixture does NOT store pre-association tracklet data. The harness cannot re-run association from a `midline_fixtures.npz` fixture without additional data.

### Pattern 3: Staged Sweep with Carry-Forward

**What:** Run parameter sweeps sequentially by priority. Best value from stage N becomes the fixed baseline for stage N+1.

```python
# Pseudocode for tune_association.py
baseline_config = {"ray_distance_threshold": 0.03, "score_min": 0.3, ...}

# Stage 1: sweep ray_distance_threshold
best_rdt = sweep_single_param("ray_distance_threshold", RANGES["ray_distance_threshold"], baseline_config)
baseline_config["ray_distance_threshold"] = best_rdt

# Stage 2: sweep score_min with updated baseline
best_sm = sweep_single_param("score_min", RANGES["score_min"], baseline_config)
baseline_config["score_min"] = best_sm
# ... and so on
```

### Pattern 4: Tier 2 Deferred to Top-N Only

**What:** Skip `compute_tier2` during the grid sweep for speed. Run full Tier 1+2 only for top-N candidates.

```python
# In run_evaluation — add skip_tier2=False parameter, or handle via top-N loop in script
# OR: run_evaluation always computes tier2, but sweep calls a "fast" variant
# Decision for planner: cleanest way to expose tier2 skipping
```

Note: The existing `run_evaluation()` always computes Tier 2. The planner should decide whether to add a `skip_tier2: bool = False` flag to `run_evaluation()`, or whether the sweep script runs a trimmed path inline.

### Pattern 5: Association-Level Metrics (Supplement to Tier 1)

**What:** Print additional metrics about association quality that `run_evaluation()` doesn't currently compute. These require accessing the fixture's MidlineSet structure directly.

From analysis doc Section 5, the recommended association-level metrics are:

```
Metric 2: Multi-camera yield
  For each frame: count fish with ≥2, ≥3, ≥4 cameras in MidlineSet
  → computed from fixture.frames (post-load, pre-reconstruction)

Metric 5: Singleton rate
  singleton_groups / total_groups
  → requires access to TrackletGroup structure, NOT available in NPZ fixture
  → can be approximated from MidlineSet: fish with only 1 camera = singleton

Metric 6: Cluster consistency score
  Mean pairwise affinity score per cluster (TrackletGroup.confidence)
  → NOT available in NPZ fixture (confidence is stored in TrackletGroup, not fixture)
```

**Critical finding:** Metrics 5 (singleton rate) and 6 (cluster consistency) require `TrackletGroup` objects, which are NOT serialized in the `midline_fixtures.npz` format. The fixture only contains `Midline2D` objects keyed by `(fish_id, camera_id)`. Singleton rate CAN be approximated from the fixture (any fish_id with only 1 camera entry = singleton), but cluster confidence cannot be recovered.

The planner should scope association-level metrics to what is computable from the fixture: multi-camera yield distribution and approximate singleton rate.

### Anti-Patterns to Avoid

- **Re-running the full pipeline for each sweep iteration:** The full pipeline requires GPU, video files, and calibration LUTs. The sweep must use the existing fixture-based harness.
- **Adding new metrics classes or schema changes:** Locked out by CONTEXT.md — no new `EvalResults` fields or JSON schema changes.
- **Mutating `AssociationConfig`:** It is `frozen=True`. Use `dataclasses.replace()`.
- **Using `ghost_penalty_enabled` toggle field as the ablation mechanism:** CONTEXT.md says to use large `ghost_pixel_threshold` value (10000) instead. No new field needed.

## Common Pitfalls

### Pitfall 1: Fixture Contains Post-Association Data — Cannot Re-Associate

**What goes wrong:** Developer adds `association_overrides` to `run_evaluation()` expecting to re-run the SPECSEED association pipeline with different parameters. But the fixture stores `fish_id` values that are already cross-camera global identities. There is no per-camera tracklet data in the NPZ.

**Why it happens:** The fixture format (`midline_fixtures.npz`) was designed to capture the result of the full pipeline (detection → tracking → association → midline extraction), not the inputs to association. `fish_id` in the fixture keys reflects post-association identity.

**How to avoid:** Before implementing `association_overrides` in the harness, clarify what it actually patches. Options:
1. `association_overrides` is a placeholder that does nothing meaningful in the fixture-based harness (association already happened when the fixture was generated). The sweep instead creates fixtures from different pipeline runs — but this is not what CONTEXT.md describes.
2. The tune_association.py script uses a different fixture type that stores pre-association data (raw tracklets per camera). This would require a new fixture format.
3. The sweep script reads the existing fixture and measures association quality from the fixture's MidlineSet structure (multi-camera yield per fish_id), without re-running association. The "overrides" passed to `run_evaluation()` only affect reconstruction parameters — and the association quality comes from how the fixture was generated.

**Recommendation for planner:** Review CONTEXT.md intent carefully. The most likely intent is option 3: the fixture was captured with the current (poorly-tuned) association parameters, and the tuning loop generates new fixtures by running the pipeline with different parameters each time, then calling `run_evaluation()` on each fixture. The `association_overrides` parameter in `run_evaluation()` may then be unnecessary or may be repurposed to control reconstruction filtering.

Alternatively, the script may need to call the association stage directly on pre-captured tracklet data stored in a separate fixture format. This would be a larger scope addition.

### Pitfall 2: `AssociationConfig` is Frozen — Cannot Mutate

**What goes wrong:** Sweep code does `config.ray_distance_threshold = 0.05` and gets `dataclasses.FrozenInstanceError`.

**Why it happens:** All config dataclasses in `engine/config.py` are `frozen=True`.

**How to avoid:** Always use `dataclasses.replace(config, **overrides)` to create a new instance. This is the same pattern used in all existing tests and scripts.

### Pitfall 3: Protocol Mismatch When Adding New AssociationConfig Fields

**What goes wrong:** Developer adds a new field (e.g., `ghost_penalty_enabled: bool = True`) to `AssociationConfig` but forgets to add it to `AssociationConfigLike` protocol in `scoring.py` or `ClusteringConfigLike` in `clustering.py`.

**Why it happens:** The core/ modules use structural protocols to avoid importing engine/. The protocols must be manually kept in sync with the dataclass.

**How to avoid:** When adding any new field to `AssociationConfig`:
1. Check if the field is consumed by `scoring.py` → add to `AssociationConfigLike`
2. Check if the field is consumed by `clustering.py` → add to `ClusteringConfigLike`
3. Check if the field is consumed by `refinement.py` → add to `RefinementConfigLike`

Since CONTEXT.md says no new `ghost_penalty_enabled` field is needed (use large threshold value instead), this pitfall should not arise if the ghost ablation is done via large `ghost_pixel_threshold`.

### Pitfall 4: Tier 2 Runtime Dominates Sweep

**What goes wrong:** Sweep over N×M parameter combinations × 15 frames × Tier 2 leave-one-out (13 cameras × N fish = ~117 additional reconstruction calls per frame) takes hours.

**Why it happens:** Tier 2 scales as O(cameras × fish × frames). At 13 cameras, 9 fish, 15 frames, that's ~1755 additional `reconstruct_frame` calls per parameter set.

**How to avoid:** Per locked decision, skip Tier 2 during the main sweep. Only run full Tier 1+2 for top-N candidates. The planner should ensure this is implemented either via a `skip_tier2` flag on `run_evaluation()` or by calling a separate Tier-2-only path.

### Pitfall 5: Score Interpretation Inversion

**What goes wrong:** Higher `score` composite value is treated as "better" when the sweep's scoring function uses "lower is better" (like `tune_threshold.py`).

**Why it happens:** `tune_threshold.py` uses a penalty-based composite score where lower = better. The association tuning script may use a different metric direction (higher yield = better).

**How to avoid:** Define the composite score for the association sweep explicitly. Likely: `primary_metric = -fish_reconstructed/fish_available` (negated so lower = better for consistent sorting), with secondary tiebreaker on `overall_mean_px`.

### Ghost Penalty Ablation (no code change needed)

```python
# Use a very large ghost_pixel_threshold to effectively disable ghost penalty
ghost_ablation_overrides = {"ghost_pixel_threshold": 10000.0}
```

This works because `score_tracklet_pair()` in `scoring.py` applies the ghost penalty only for detections within `ghost_pixel_threshold` pixels. Setting it to 10000 means all detections within 10000px "support" the ghost point, making ghost_ratio ≈ 0 and penalty ≈ 1.0 (no penalty).

### Multi-Camera Yield from Fixture (association-level metric)

```python
# Computable from fixture.frames without additional data
def compute_multicam_yield(frames: tuple) -> dict:
    """Count fish by camera count across all frames."""
    counts = {1: 0, 2: 0, 3: 0, 4: 0}
    for midline_set in frames:
        for fish_id, cam_map in midline_set.items():
            n_cams = len(cam_map)
            counts[min(n_cams, 4)] = counts.get(min(n_cams, 4), 0) + 1
    return counts
```

## Open Questions

1. **What does `association_overrides` actually do in `run_evaluation()`?**
   - What we know: The fixture stores post-association data. The harness cannot re-run SPECSEED from fixture data. But CONTEXT.md says "runs through `run_evaluation()` for each config."
   - What's unclear: Does the sweep generate a new fixture per config (by running the full pipeline), or does it use a single fixture and vary only reconstruction parameters? Or is there a pre-association fixture format not yet created?
   - Recommendation: **The planner must resolve this before writing task breakdowns.** Most likely interpretation: `tune_association.py` will run the full pipeline (or just the association stage) with different `AssociationConfig` values each time, then call `run_evaluation()` on the resulting fixture. This would require either (a) a cached set of pre-association tracklet data that can be re-associated quickly, or (b) accepting that the full pipeline runs N times (slow but correct). Alternatively: `run_evaluation()` accepts an already-associated `MidlineSet` directly (not a fixture path), and `tune_association.py` re-runs association itself and passes results to the harness.

2. **Should `run_evaluation()` accept a `MidlineSet` directly, or only a fixture path?**
   - What we know: Current signature takes `fixture_path: Path | str`.
   - What's unclear: If the sweep re-runs association to produce `MidlineSet` objects in memory, wrapping them in a fixture file for each sweep iteration is wasteful.
   - Recommendation: The planner should consider whether `run_evaluation()` should be refactored to accept an in-memory `MidlineFixture` or `MidlineSet`, or whether a thin wrapper function is more appropriate.

3. **Can Tier 2 be skipped without modifying `run_evaluation()`?**
   - What we know: `run_evaluation()` always computes Tier 2. Tier 2 is expensive for large sweeps.
   - What's unclear: Adding `skip_tier2=False` is the cleanest approach but requires a harness signature change beyond what CONTEXT.md specifies.
   - Recommendation: Add `skip_tier2: bool = False` to `run_evaluation()` alongside `association_overrides`. This is within the spirit of "extend with `association_overrides` dict parameter" and is needed for practical sweep speed.
