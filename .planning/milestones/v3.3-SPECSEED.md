# Milestone Seed: v3.3 Chunk Processing

**Status:** SEED (pre-requirements)
**Predecessor:** v3.2 Evaluation Ecosystem (Phases 46-50)
**Created:** 2026-03-03

## Problem

The cross-view association stage (Leiden clustering on pairwise tracklet scores) scales O(T²) in tracklet count per camera pair. Longer videos produce more tracking fragments, which makes association both slower and less reliable — denser affinity graphs increase the risk of incorrect clustering or Leiden non-convergence (which has no timeout and can hang indefinitely). A single pathological segment with heavy fragmentation can poison the global solution for the entire video.

Memory is not the primary constraint. With the pose estimation backend (no segmentation masks), a 1000-frame chunk uses ~90 MB and a full 30-minute video (54K frames) uses ~5 GB — comfortable on any workstation. The motivation for chunking is association robustness and bounded problem complexity, not OOM avoidance.

### Memory Estimates (pose estimation backend, 13 cameras, 9 fish)

| Stage | Per 1000 Frames | Full 30-min (54K frames) |
|-------|----------------|--------------------------|
| Detection | ~18 MB | ~1.0 GB |
| Tracking | ~1 MB | ~1 MB (global) |
| Association | ~0.25 MB | ~0.25 MB (global) |
| Midline (no masks) | ~65 MB | ~3.5 GB |
| Reconstruction | ~5 MB | ~0.3 GB |
| **Total** | **~90 MB** | **~4.9 GB** |

Note: The segmentation backend stores ~40 KB masks per detection, which would push Stage 4 to ~2.6 GB per 1000 frames. If segmentation is ever used for long videos, chunking provides memory relief as a secondary benefit.

## Solution

Process videos in fixed-size temporal chunks. Each chunk runs the full 5-stage pipeline independently — association runs unchanged per chunk, producing chunk-local fish IDs. A lightweight post-chunk identity stitching step maps chunk-local IDs to globally consistent IDs using track ID continuity from the tracking carry-forward. A chunk orchestrator manages the loop, handoff state, and HDF5 output accumulation.

## Architecture

### Three-Layer Model

Chunking introduces a new layer above PosePipeline:

```
ChunkOrchestrator (run-level concerns)
  ├── Manages chunk loop, frame windowing, handoff state
  ├── Owns HDF5 output accumulation (run-level, not per-chunk)
  ├── Owns identity stitching across chunks
  └── Owns progress reporting
  │
  └── PosePipeline (chunk-level concerns, unchanged)
        ├── Runs 5 stages on whatever frames it receives
        ├── Fires lifecycle events (PipelineStart, StageComplete, etc.)
        └── Observers handle chunk-level side effects (timing, logging)
              │
              └── Stages (stage-level concerns, unchanged)
                    └── Process every frame they receive, 0-indexed locally
```

Key principle: **PosePipeline and its stages do not know they are being run in chunks.** The orchestrator presents each chunk as a complete short video. Stages process every frame they receive. The orchestrator handles global frame indexing, output accumulation, and cross-chunk state.

### Frame Source Abstraction

Currently DetectionStage and MidlineStage each create their own `VideoSet` internally and iterate all frames. This couples stages to video I/O and makes chunking impossible without stage modifications.

**Refactor:** Extract frame sourcing from stages into an injectable dependency, passed via constructor injection (consistent with how backends and other dependencies are already wired). A frame source is an iterator yielding `(frame_idx, dict[str, ndarray])` — local frame index plus per-camera undistorted frames. Stages consume frames from the source without knowing where they come from or how undistortion was applied.

`VideoSet` already supports optional undistortion — the frame source wraps it so that undistortion is a frame-source concern, not a stage concern. No downstream task needs distorted frames. Stages no longer need `video_dir` or `calibration_path` for frame access.

The orchestrator creates a windowed frame source for each chunk (a `VideoSet` wrapper that seeks to the chunk start and yields only chunk_size frames, 0-indexed locally). For non-chunked runs, it creates a full-video frame source. The pipeline and stages are identical in both cases.

This also eliminates `stop_frame` from pipeline config — it was a testing convenience for "process fewer frames than the video contains," which is now naturally expressed as "give the pipeline a frame source that yields N frames."

### Chunk Orchestrator

```
for each chunk [start_frame, end_frame):
    1. Create windowed frame source for this chunk
    2. Inject carry-forward state from prior chunk (if any)
    3. Run PosePipeline (stages see a short video, 0-indexed)
    4. Stitch chunk-local fish IDs to global IDs
    5. Flush 3D midlines to HDF5 (with global frame indices)
    6. Extract and serialize ChunkHandoff (atomic write)
    7. Release chunk data from memory
```

When `chunk_size` is null/0, the orchestrator degenerates to a single iteration with a full-video frame source — identical to current behavior.

### ChunkHandoff

A single frozen dataclass carries all cross-chunk state:

```python
@dataclass(frozen=True)
class ChunkHandoff:
    # Per-camera OC-SORT tracker state (absorbs existing CarryForward)
    tracker_state: dict[str, OcSortState]

    # Identity mapping: cam_id → {track_id → fish_id}
    identity_map: dict[str, dict[int, int]]
```

This replaces the existing `CarryForward` dataclass, which only carried tracker state. The unified handoff ensures tracking continuity *and* identity continuity share a single serialization boundary.

`FishState3D` (3D position/velocity for re-identifying lost tracks) and `cumulative_frames` (cross-chunk frame count for reliability weighting) were considered and deferred — see Key Design Decisions.

### Identity Stitching

Each chunk runs full association independently, producing chunk-local fish IDs. The orchestrator maps these to global IDs using track ID continuity:

1. From the prior chunk's handoff, we have `identity_map`: `(cam_id, track_id) → global_fish_id`.
2. OC-SORT carry-forward preserves track IDs across chunk boundaries, so many tracklets in the new chunk share track IDs with the prior chunk.
3. For each tracklet group produced by association, check its constituent tracklets against `identity_map`. If any tracklet has a known `(cam_id, track_id)`, assign the group that global fish ID.
4. Groups with no matching track IDs are new fish — assign fresh global IDs.
5. Build the updated `identity_map` from this chunk's tracklet groups for the next handoff.

This is lightweight — it operates on tracklet group metadata, not frame data. For stable tracking (most chunks after the first), every group maps to a known fish and no new IDs are needed.

### Per-Chunk Association (Unchanged)

Association runs identically to current non-chunked behavior within each chunk. No seed phase, no special chunk awareness. Each chunk has a small, bounded tracklet count (~9-15 per camera for 1000 frames), which is the same problem size association already handles well.

**Potential extension — seed + resolve:** If identity stitching proves insufficient (e.g., frequent track ID breaks cause too many "new fish" assignments), association could be enhanced with a seed phase that pre-assigns known identities before clustering. This would reduce Leiden's workload and improve continuity. Deferred because per-chunk association with post-hoc stitching is simpler and likely sufficient.

### Observer Behavior

With chunking, PosePipeline fires its lifecycle events (PipelineStart, StageComplete, PipelineComplete) once per chunk. This creates a run-level vs. chunk-level distinction:

| Concern | Layer | Behavior |
|---------|-------|----------|
| HDF5 output | Orchestrator | Accumulates across chunks; not an observer |
| Handoff serialization | Orchestrator | Writes after each chunk |
| Progress reporting | Orchestrator | "Chunk 3/50, frame 2000-2999" |
| Per-stage timing | Observer | Per-chunk timing (fine as-is) |
| Logging | Observer | Per-chunk logging (fine as-is) |
| Diagnostic observer | N/A | Disabled; chunk mode and diagnostic mode are mutually exclusive |
| HDF5 observer | Disabled | Orchestrator handles HDF5; observer must not also write |

The HDF5 observer (currently writes at PipelineComplete) must be disabled when the orchestrator is active. The orchestrator owns HDF5 output as a run-level concern.

### HDF5 Output

`Midline3DWriter` already supports incremental chunked writes with a 1000-frame buffer. The orchestrator flushes 3D midlines after each chunk completes (applying the global frame offset), so output accumulates on disk and is never fully in memory.

### Handoff Serialization

A single `handoff.pkl` is written atomically (write to temp file, then rename) after each chunk completes. Only the most recent handoff is kept — prior chunks' handoffs are overwritten. This provides a natural resumption point for future work (restart logic is out of scope for this milestone, but the data is on disk).

## Configuration

```yaml
pipeline:
  chunk_size: 1000  # frames per chunk; 0 or null = full-video (no chunking)
```

- `chunk_size: 0` or `chunk_size: null` falls back to current full-video processing with no behavioral change.
- A warning is emitted if `chunk_size < 100` frames, since very small chunks degrade association quality due to insufficient temporal evidence (100 matches `t_saturate`).
- `stop_frame` is removed from pipeline config. Frame windowing is now a frame-source concern managed by the orchestrator.

## Mutual Exclusivity with Diagnostic Mode

Chunk processing and diagnostic mode are mutually exclusive for this milestone. Diagnostic mode is a development/tuning tool that operates on short clips (effectively a single chunk). The diagnostic observer, per-stage pickle caches, and evaluation tooling are unaffected — they attach only in diagnostic mode, which does not use chunk orchestration.

## Phasing

### Phase 1: Frame Source Refactor
Extract video I/O from DetectionStage and MidlineStage into an injectable frame source. Remove `stop_frame` from pipeline config. All existing behavior preserved — stages just receive frames from the source instead of opening videos themselves. This phase delivers value independently (cleaner architecture, easier testing) and is a prerequisite for chunk orchestration.

### Phase 2: Chunk Orchestrator and Handoff
Implement ChunkOrchestrator, ChunkHandoff, windowed frame source, per-chunk HDF5 flush, atomic handoff serialization, config field, and validation. Identity stitching uses track ID continuity from the handoff.

### Phase 3: Integration and Validation
Wire orchestrator into `aquapose run`, disable HDF5 observer when chunking is active, validate that chunked output matches non-chunked output for the same video.

## Scope Boundaries

### In Scope
- Frame source abstraction replacing direct VideoSet usage in stages
- Removal of `stop_frame` from pipeline config
- ChunkOrchestrator with fixed-size temporal chunking
- ChunkHandoff replacing CarryForward (tracker state + identity map)
- Post-chunk identity stitching via track ID continuity
- Per-chunk HDF5 flush via existing Midline3DWriter
- HDF5 observer disabled under chunk mode
- Atomic handoff serialization to disk
- Config field (`chunk_size`) with null/0 fallback to full-video mode
- Warning for small chunk sizes
- Validation: chunked output matches non-chunked output

### Out of Scope
- Resumption from crash (handoff is serialized, but restart logic is deferred)
- Adaptive chunk sizing
- Parallel chunk processing
- Diagnostic mode compatibility (mutually exclusive)
- Temporal overlap at chunk boundaries (hard boundaries + carry-forward suffice)
- Tracklet splitting (no mechanism exists; fragment merging handles the inverse)
- Intermediate stage output persistence (only final 3D midlines flushed per chunk)
- Seed + resolve association (potential extension if identity stitching proves insufficient)

## Key Design Decisions

| Decision | Rationale |
|----------|-----------|
| Per-chunk association (not global) | Global association is O(T²) in tracklets and risks incorrect clustering or non-convergence on long videos. Per-chunk bounds the problem size and isolates failures. |
| Full association per chunk + post-hoc stitching (not seed + resolve) | With 1000-frame chunks, each chunk's tracklet count is similar to current short-clip runs — association already handles this well. Seed + resolve adds complexity (Leiden doesn't support pre-assigned nodes) for a problem that doesn't exist at this chunk size. Seed + resolve is a potential extension if stitching proves insufficient. |
| Identity stitching via track ID continuity | OC-SORT carry-forward preserves track IDs across chunk boundaries. Matching tracklet groups to known IDs via shared track IDs is lightweight and leverages existing infrastructure. |
| Frame source abstraction | Decouples stages from video I/O. Enables chunk windowing without stage modifications. Eliminates `stop_frame` as a pipeline config concern. Improves testability. |
| Orchestrator owns HDF5 output | HDF5 output is a run-level concern (accumulates across chunks). The per-chunk HDF5 observer would fire incorrectly. Orchestrator manages the single output file. |
| Hard chunk boundaries (no overlap) | Temporal overlap adds complexity (duplicate detection, merge logic). Carry-forward state provides continuity without it. Overlap can be added later if needed. |
| Atomic handoff write | Prevents corrupted handoff if crash occurs mid-write. Trivial to implement (temp file + rename). |
| Diagnostic mode exclusion | Keeps scope bounded. Diagnostic mode serves a different purpose (short-clip tuning) and doesn't need chunk support. |
| No FishState3D in handoff | 3D re-identification of lost tracks is premature. Unmatched tracklets go through normal Leiden clustering within the chunk. Add 3D proximity matching later if re-ID failures are observed in practice. |
| No cumulative frame count in handoff | The overlap reliability weight (`effective_saturate = min(t_saturate, frame_count)`) only matters when `frame_count < t_saturate` (100). With 1000-frame chunks this can't happen. Each chunk's association scores only that chunk's tracklets with that chunk's evidence, which is correct behavior. The minimum chunk size warning (≥100 frames) guards against the edge case. |

## Resolved Design Questions

1. **Chunk size default**: 1000 frames (~33 sec at 30fps). Balances association quality (10x `t_saturate`) against bounded fragment count.
2. **Warning threshold**: 100 frames (matches `t_saturate`). Below this, association lacks sufficient temporal evidence for reliable scoring.
3. **Frame source interface**: Constructor injection, consistent with how backends and other dependencies are already wired into stages. Frame source yields undistorted frames — undistortion is a frame-source concern, not a stage concern.
