# v3.6 Model Iteration & QA

## Goal

Run the pseudo-label retraining loop end-to-end, starting from raw manual annotations through the new data store, to produce demonstrably better OBB detection and pose estimation models. Fix bugs encountered along the way. All data and models tracked through the Phase 68 store/lineage system from the start.

## Context

- **Models in scope**: OBB detection and pose estimation only (segmentation excluded due to unresolved cross-camera orientation problem)
- **Source data**: ~50 full-frame COCO-JSON annotated images yielding ~322 pose crops. Original labels created when tank was freshly cleaned — current conditions include algae on tank walls causing persistent false positives.
- **Iteration target**: 1-2 rounds of the pseudo-label loop
- **Run data**: 12 five-minute clips (one camera each, synchronized), ~9450 frames at 30fps per clip, 12 cameras total
- **Run lengths**: Short iteration runs (~1 min / ~1800 frames / ~9 chunks, ~12 min runtime) for fast loop turnaround. Full 5-min validation run at the end for final confirmation and showcase outputs.
- **Elastic augmentation**: Include in round 1 (few curved-fish examples in manual set). Evaluate whether needed in round 2 — pseudo-label curvature diversity sampling may make it redundant.
- **Calibration**: Current calibration is adequate (~2px reprojection error). Calibration refinement deferred to v3.7.
- **Workflow style**: Guided manual execution. Phases contain explicit user-executed steps with decision checkpoints. Agent prepares detailed game plan; user runs commands and reports results; agent confirms and adjusts.

## Validation Approach

- **Pipeline metrics**: Singleton rate, reprojection error, track continuity, detection count per frame (via `aquapose eval`)
- **Visual inspection**: Overlay videos showing reprojected 3D midlines, spot-checking for false positives, missed detections, and tracking errors
- **Training metrics**: Ultralytics training/val loss curves, mAP, OKS (via `aquapose train compare`)
- **Before/after comparison**: Baseline metrics from round 0 models vs each subsequent round on identical clips

## Phases

### Phase 70: Metrics & Comparison Infrastructure

**Goal**: Extend evaluation metrics before the iteration loop starts so that all rounds are measured consistently. Diagnostic caches preserve raw data, so these metrics can be computed post-hoc — but having the analysis code ready avoids scrambling between rounds.

Additions to existing evaluators:
- **Reprojection error percentiles** (p50, p90, p95) — add to reconstruction evaluator alongside existing mean/max
- **Midline confidence percentiles** (p10, p50, p90) — add to midline evaluator alongside existing mean/std
- **Camera count per-reconstruction percentiles** — add to association evaluator (already has `camera_distribution` histogram, add p50/p90 summary)

New analysis code (operates on cached data):
- **Per-keypoint reprojection error** — breakdown by body point index (0=head through 14=tail), reported as mean + p90 per keypoint. New section in reconstruction evaluator or standalone analysis function.
- **Curvature-stratified reconstruction quality** — compute curvature from 3D spline, bin reconstructions into curvature quantiles, report reprojection error per bin. New analysis function.
- **3D track fragmentation** — post-association track count, gap count/duration stats, and continuity ratio (frames with reconstruction / frames where fish is visible in >=3 cameras). New section in reconstruction or association evaluator.

All metrics surfaced in `aquapose eval` text/JSON output.

### Phase 71: Data Store Bootstrap

**Goal**: All existing manual annotations imported into the Phase 68 data store, baseline OBB and pose models trained and registered.

Steps:
1. Convert COCO-JSON full-frame annotations to YOLO-OBB format (full-frame) and YOLO-pose format (crop-space)
2. Import both into the data store as `source=manual`
3. Assemble baseline training datasets (train/val split matching the existing curvature-stratified split)
4. Train baseline OBB model
5. Train baseline pose model (with elastic augmentation)
6. Register both models in the store
7. Verify: models reproduce approximately similar quality to current models (sanity check, not exact match expected)

**QA focus**: Validates the full `data convert` -> `data import` -> `data assemble` -> `train` -> model registration workflow end-to-end.

### Phase 72: Baseline Pipeline Run & Metrics

**Goal**: Establish quantitative "before" snapshot on short iteration clips using baseline models.

Steps:
1. Run pipeline in diagnostic mode on short clip (~1 min) with baseline models
2. Run `aquapose eval` to capture baseline metrics (singleton rate, reprojection error, track continuity, per-stage breakdowns)
3. Visual inspection of overlay video — document false positive rate, missed detections, obvious failure modes
4. Record baseline numbers as the benchmark for improvement

**QA focus**: Validates the full `run` -> `eval` -> `viz` workflow with new models from the store.

### Phase 73: Round 1 Pseudo-Label Generation & Retraining

**Goal**: Generate pseudo-labels from baseline run, retrain models, measure improvement.

Steps:
1. Generate OBB pseudo-labels from baseline run diagnostic caches (`pseudo-label generate`)
2. Generate pose pseudo-labels from baseline run
3. Inspect pseudo-labels visually (`pseudo-label inspect`) — confirm algae false positives are excluded, label quality is acceptable
4. Import pseudo-labels into store as `source=pseudo, round=1`
5. Assemble round 1 training datasets:
   - OBB: manual + pseudo (Source A + Source B with confidence filtering)
   - Pose: manual + elastic augmentation + pseudo (Source A with confidence filtering)
6. Train round 1 OBB model
7. Train round 1 pose model
8. Register models, run `train compare` against baseline

**Decision checkpoint**: Do training metrics show clear improvement? If not, diagnose before proceeding.

### Phase 74: Round 1 Evaluation

**Goal**: Confirm round 1 models improve pipeline-level results, not just training metrics.

Steps:
1. Re-run pipeline on same short clip with round 1 models
2. Run `aquapose eval`, compare to Phase 72 baseline
3. Visual inspection of overlay video — are false positives reduced? Are previously missed fish now detected? Track continuity improved?
4. Decision: Is there room for a second round, or are results already good?

**Decision checkpoint**: Proceed to round 2, or skip to final validation?

### Phase 75 (conditional): Round 2

**Goal**: Squeeze out additional improvement if round 1 left clear headroom.

Steps:
1. Generate pseudo-labels from round 1 run
2. Import as `source=pseudo, round=2`
3. Assemble round 2 datasets (evaluate whether elastic augmentation is still needed — check curvature distribution in pseudo-labels vs augmented set)
4. Train round 2 models
5. Re-run pipeline, evaluate, compare to rounds 0 and 1
6. Visual inspection

### Phase 76: Final Validation Run

**Goal**: Confirm results hold at full scale and produce showcase outputs.

Steps:
1. Run full 5-minute pipeline with best models from iteration
2. Full `aquapose eval` report
3. Generate overlay videos for all 12 cameras
4. Side-by-side comparison: baseline overlay vs final overlay on same time segment
5. Summary document: metrics table (round 0 vs round 1 vs round 2 vs final), key visual comparisons, known remaining limitations

**This produces the "clearly good results" artifact.**

### Bug Fixes & QA

Bugs encountered during any phase are fixed in-place rather than deferred. Expected areas of friction:
- Data store edge cases (dedup behavior, augmentation interaction with pseudo-labels)
- CLI ergonomics discovered through actual use (Phase 69 may not have caught everything)
- Pseudo-label quality issues (confidence thresholds may need tuning)
- Pipeline behavior differences between short and full-length runs
- Edge cases in frame selection / diversity sampling with real data

## Out of Scope

- Segmentation model iteration (cross-camera orientation problem unresolved)
- Calibration refinement (v3.7)
- Documentation and publication artifacts (v4.0)
- New pipeline features or architectural changes
- Full-day recording support

## Success Criteria

1. Detection false positive rate visibly reduced (algae no longer triggers detections)
2. Pipeline metrics improve over baseline (lower singleton rate, lower reprojection error, or better track continuity)
3. All data and models tracked in the store with full provenance (round, source, confidence thresholds)
4. Overlay videos show tight 3D midline reprojection with few visible errors
5. No known bugs blocking a clean hand-off of the iteration workflow to another user
