# Phase 91.1: Association Bottleneck Investigation & Remediation - Context

**Gathered:** 2026-03-11
**Status:** Ready for planning

<domain>
## Phase Boundary

Identify and fix wall-time performance bottlenecks in the v3.8 association stage. The association stage currently consumes 452s (85% of total pipeline time) per 300-frame chunk. The bottleneck is in recovery (specifically `_attempt_split_assign`), with validation as a secondary contributor. Scoring is already fast (0.7s). This phase adds instrumentation, vectorizes all scalar ray-casting loops, and fixes the timing observer.

</domain>

<decisions>
## Implementation Decisions

### Investigation approach
- Add per-step timing instrumentation to `AssociationStage.run()` (scoring, clustering, validation, recovery logged separately)
- Fix the timing observer to **append** to `timing.txt` with per-chunk timing data rather than overwriting the file
- No formal profiling (cProfile/flame graph) — evidence from run logs is sufficient: scoring 0.7s, validation ~13s, recovery ~437s

### Root cause
- Recovery's `_attempt_split_assign` recomputes ray casts from scratch for every (split_point × group) combination instead of precomputing per-frame residuals once and sweeping split points over the precomputed array
- Validation's `_compute_frame_residual_keypoints` uses per-frame per-keypoint scalar `cast_ray(1 pixel)` + scalar `ray_ray_closest_point` in Python loops
- Same antipattern in consensus/confidence recomputation (`_compute_frame_consensus`, `_compute_per_frame_confidence`)

### Remediation strategy
- **CPU numpy batching** for all scalar patterns: batch all pixels into a single `cast_ray` call per camera, then `ray_ray_closest_point_batch`. Matches existing pattern in `scoring.py`
- **Precompute-then-sweep** for split-assign: compute per-frame residuals for each (singleton, group) pair once, then sweep split points as pure numpy index arithmetic
- No GPU migration — stay with existing `.cpu().numpy()` pattern
- Vectorize ALL scalar cast_ray + ray_ray_closest_point patterns in both `validation.py` and `recovery.py`, including consensus and confidence helpers

### Performance targets
- Total association stage budget: **under 30s per 300-frame chunk** (from current 452s)
- If vectorization alone doesn't achieve this, disable split-assign as a last resort via config toggle (already has `recovery_min_segment_length` as a natural no-op)

### Fallback strategy
- Optimize first, disable split-assign only as last resort
- Split-assign is the lowest-confidence recovery path (per SPECSEED design doc) so disabling it is acceptable if needed

### Claude's Discretion
- Exact batching granularity (batch per-group vs batch all groups at once)
- Whether to add a `recovery_split_enabled` config toggle or rely on the existing `recovery_min_segment_length` (set high = disabled)
- Internal function decomposition within the vectorized implementations

</decisions>

<specifics>
## Specific Ideas

- "For the cast_ray calls, I thought we were going to calculate them once then sweep across the split points" — the design intent from the SPECSEED doc describes precomputing residuals, but the implementation recomputes from scratch per split point
- Timing observer should append per-chunk data, not overwrite — user wants cumulative timing across chunks in a single file

</specifics>

<code_context>
## Existing Code Insights

### Reusable Assets
- `ray_ray_closest_point_batch` in `scoring.py`: Already-vectorized batch ray-ray distance function — validation and recovery should use this instead of scalar `ray_ray_closest_point`
- `_batch_score_frames_kpt` in `scoring.py`: Reference pattern for batched keypoint extraction, flattened cast_ray calls, and reduceat aggregation
- `ForwardLUT.cast_ray`: Accepts batched pixel tensors `(N, 2)` — already supports the batch pattern, just being called with N=1 repeatedly

### Established Patterns
- Scoring already demonstrates the correct pattern: extract keypoints → build confidence mask → flatten valid pairs → single `cast_ray` call → `ray_ray_closest_point_batch` → reduceat aggregation
- `.cpu().numpy().astype(np.float64)` conversion chain after `cast_ray` is the standard flow

### Integration Points
- `AssociationStage.run()` in `stage.py`: Timing instrumentation goes here (wrap each step with `time.perf_counter()`)
- Timing observer: `src/aquapose/engine/` — needs the append-vs-overwrite fix
- `validation.py`: 4 functions to vectorize (`_compute_frame_residual_keypoints`, `_compute_frame_residual_centroid`, `_compute_frame_consensus`, `_compute_per_frame_confidence`)
- `recovery.py`: 3 functions to vectorize (`_score_singleton_against_group`, `_triangulate_group_keypoints_for_frame`, `_attempt_split_assign` precompute restructuring)

</code_context>
