# Phase 78.1: OBB & Pose Production Retrain - Context

**Gathered:** 2026-03-10
**Status:** Ready for planning

<domain>
## Phase Boundary

Retrain OBB detector and pose model with corrected pseudo-labels in a proper all-source train/val split, with more epochs targeting white-wall recall improvement. This is a terminal retrain producing production models for the v3.7 tracker milestone. No further pseudo-label rounds planned.

</domain>

<decisions>
## Implementation Decisions

### Data composition
- All-source stratified val split — val drawn from ALL sources (manual + corrected pseudo-labels), stratified by camera/curvature. Not manual-only val.
- Curated pseudo-labels only — these are the only labels currently in the data store (upsert import replaced uncurated)
- No special white-wall data sourcing — rely on more epochs to improve recall on underrepresented cases
- Elastic augmentation for pose model (curated+aug pattern from Round 1 winner: 5 variants/image)

### Training parameters
- OBB: 300 epochs, patience=50 (early stopping), imgsz=640, YOLO26n-obb architecture
- Pose: 300 epochs, patience=50, imgsz=320 with rect=True (rectangular batches), YOLO26n-pose architecture
- No architecture changes — this is a data/epochs improvement, not an architecture experiment

### Evaluation criteria
- Compare against Round 1 winners: OBB curated (run_20260309_120659), Pose curated+aug (run_20260309_152248)
- Primary metric: mAP50-95 on the new all-source stratified val set
- White-wall recall: visual spot-check using `scripts/investigate_occlusion.py` on known white-wall frames with new models
- Post-training: re-run investigation script on first 20 seconds of full-frame video to produce visualization with new models

### Deployment
- Update YH project config.yaml to point to new best model paths
- Round 1 model files kept in runs directory but not configured — manual revert if needed
- Terminal retrain: no pseudo-label regeneration, models feed directly into Phase 80+

### Training execution handoff
- Agent prepares everything (dataset assembly, augmentation, config) then provides exact CLI training commands
- User runs training manually (agentic training causes more problems than it solves)
- Agent waits for user to report training completion, then resumes with evaluation and deployment

### Claude's Discretion
- Dataset assembly query parameters and stratification details
- Augmentation parameters (use Round 1 defaults: 5-15 degree angle, 4 variants/image)
- Exact val split ratio and stratification criteria
- Investigation script parameters for the post-training visualization

</decisions>

## Existing Code Insights

### Reusable Assets
- `SampleStore` (training/store.py): SQLite-backed data management with import/assemble/exclude/include
- `data assemble` CLI: Assembles YOLO datasets from store with query filters
- `train obb` / `train pose` CLI: Full training lifecycle with run directory, config snapshot, model registration
- `train compare` CLI: Side-by-side model comparison
- `elastic_deform.py`: Elastic augmentation for pose (C-curve, S-curve variants)
- `scripts/investigate_occlusion.py`: Investigation script with full-frame support, reads model paths from project config

### Established Patterns
- Training workflow: assemble dataset → train → compare → update config (established in Phase 73)
- Pose model: `rect=True` + `imgsz=320` (rectangular batches, not square crops)
- Run identity: timestamp-based (`run_YYYYMMDD_HHMMSS`)
- Model registration: best weights path recorded in run metadata

### Integration Points
- Project config (`~/aquapose/projects/YH/config.yaml`): model weight paths consumed by pipeline and investigation script
- Data store (`~/aquapose/projects/YH/`): curated pseudo-labels already imported
