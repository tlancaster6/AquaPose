  <files>.planning/phases/38-stabilization-and-tech-debt-cleanup/38-DEAD-CODE-REPORT.md</files>
  <action>
    Use `ast.parse()` and `ast.walk()` (Python's ast module) to build a complete import graph for the project. Analyze every `.py` file under `src/aquapose/`.

    **For each file in the legacy directories** (`reconstruction/`, `segmentation/`, `tracking/`, `visualization/`):

    1. **Identify all importers:** Which files import from this module? List every importing file.
    2. **Classify the file:**
       - **Canonical implementation:** Contains unique logic that is imported by `core/` or `engine/` code. Cannot be deleted without migration.
       - **Thin wrapper / re-export:** Only re-exports symbols from another module. Candidate for inlining the imports at the call sites and deleting the wrapper.
       - **Dead code:** Not imported by any other file. Candidate for deletion.
    3. **For canonical implementations:** Note what would need to happen to migrate the code (which consumers would need import updates).
    4. **For thin wrappers:** Note which call sites import from them and what the direct import path would be.

    **Report format (38-DEAD-CODE-REPORT.md):**

    ```markdown
    # Dead Code Analysis Report — Phase 38

    ## Methodology
    - ast.parse() import graph analysis of all src/aquapose/**/*.py files
    - Date: {date}

    ## Summary
    - Total legacy files analyzed: N
    - Canonical implementations (active, cannot delete): N
    - Thin wrappers (can inline and delete): N
    - Dead code (can delete): N

    ## Detailed Findings

    ### reconstruction/
    #### reconstruction/midline.py — CANONICAL
    **Imported by:** core/midline/backends/segmentation.py, ...
    **Contains:** [brief description of unique logic]
    **Recommendation:** Keep or migrate to core/reconstruction/

    ### segmentation/
    ...

    ## Recommended Actions
    1. [action] — [justification]
    2. [action] — [justification]
    ```
