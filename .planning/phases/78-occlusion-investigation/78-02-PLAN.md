Run the script on camera `e3v831e` targeting the ~13-14 second mark (frames ~390-420 at 30fps) plus surrounding context. Use the full 20-second clip for confidence sweep:

If the script errors, fix bugs and re-run. Common issues: CUDA OOM (reduce batch), path resolution, missing camera ID in video dir.

**Step 2: Analyze the annotated video and per-frame statistics**

Review the output video frame-by-frame (or use the per-frame JSON stats) to characterize:

1. **OBB behavior during occlusion:**
   - Do OBB boxes merge when fish overlap? How many consecutive frames?
   - Do boxes split correctly when fish separate?
   - Are there missed detections during occlusion?
   - Does confidence drop during occlusion?

2. **Keypoint behavior during occlusion:**
   - Do keypoints jump between fish (identity jump)? Count frames where this happens.
   - How does per-keypoint confidence behave? Which keypoints are most affected?
   - Are secondary pose instances detected during occlusion? How frequently?
   - Do secondary instances capture the occluded fish's pose?

3. **Apply go/no-go criteria from CONTEXT.md:**
   - Count frames with keypoint identity jumps during occlusion windows
   - Calculate percentage: (glitch frames / occlusion frames)
   - No-go if >20% of occlusion frames have single-frame keypoint glitches
   - OBB merging: only flag if >5 consecutive frames

4. **Analyze confidence sweep results:**
   - Read the confidence_sweep.md table
   - Identify the threshold where false positives drop off without losing true detections
   - Consider the tradeoff for tracker design (lower threshold = more data but noisier)

**Step 3: Save key frame screenshots**

Extract 3-6 key frames from the annotated video as PNG screenshots in `.planning/phases/78-occlusion-investigation/screenshots/`:
