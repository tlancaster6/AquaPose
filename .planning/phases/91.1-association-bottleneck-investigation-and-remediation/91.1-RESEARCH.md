## Summary

The association stage consumes 452s per 300-frame chunk (85% of total pipeline time). The root cause is two layers of nested Python loops that call `cast_ray(1 pixel)` and the scalar `ray_ray_closest_point()` one pair at a time. Scoring already demonstrates the correct pattern — a single `cast_ray` call with all valid keypoints batched together, then `ray_ray_closest_point_batch` — but `validation.py` and `recovery.py` were never updated to follow this pattern.

The secondary fix is the `_attempt_split_assign` algorithm. It currently recomputes `_score_singleton_against_group()` (which itself has nested loops) for every split point × group combination in a double loop. Since `_score_singleton_against_group` does per-frame residual computation, the total cost scales as O(split_points × groups × frames × keypoints) in Python loop iterations. The fix is to precompute a `(n_groups, n_frames)` residual array once per singleton and then sweep split points as cheap numpy index arithmetic over the precomputed array.

The third fix is the `TimingObserver` which calls `write_text()` (overwrite mode) on every pipeline completion. For a multi-chunk run, each chunk's timing report overwrites the previous one. It should `open(path, "a")` to accumulate chunk reports. Per-step timing instrumentation in `AssociationStage.run()` is also needed so the planner can confirm post-optimization budget compliance.

**Primary recommendation:** Vectorize all `cast_ray(1 pixel)` + scalar `ray_ray_closest_point` occurrences in `validation.py` and `recovery.py` using the existing `scoring.py` batch pattern, then restructure `_attempt_split_assign` to precompute residuals before sweeping split points.

### Pattern 1: Batch cast_ray + ray_ray_closest_point_batch

**What:** Replace N individual `cast_ray(1 pixel)` calls with one `cast_ray(N pixels)` call, then replace N scalar `ray_ray_closest_point()` calls with one `ray_ray_closest_point_batch(N pairs)`.

**When to use:** Whenever iterating over frames, keypoints, or cameras and calling `cast_ray` inside the loop.

**Reference implementation** (from `scoring.py` `_batch_score_frames_kpt`):

```python
# Source: src/aquapose/core/association/scoring.py

# Step 1: collect all valid pixels across frames into a flat (M, 2) array
pixels_a = kpts_a[valid].astype(np.float64)  # (M, 2)
pixels_b = kpts_b[valid].astype(np.float64)  # (M, 2)

# Step 2: single cast_ray call per camera
pix_a_t = torch.tensor(pixels_a, dtype=torch.float32)
pix_b_t = torch.tensor(pixels_b, dtype=torch.float32)
origins_a, dirs_a = lut_a.cast_ray(pix_a_t)
origins_b, dirs_b = lut_b.cast_ray(pix_b_t)

# Step 3: convert to float64 numpy
oa = origins_a.cpu().numpy().astype(np.float64)
da = dirs_a.cpu().numpy().astype(np.float64)
ob = origins_b.cpu().numpy().astype(np.float64)
db = dirs_b.cpu().numpy().astype(np.float64)

# Step 4: vectorized ray-ray distances for all M pairs
dists = ray_ray_closest_point_batch(oa, da, ob, db)  # (M,)

# Step 5: scatter back to per-frame means using reduceat
active_n_valid = n_valid[active_mask]          # (A,) count per active frame
offsets = np.concatenate([[0], np.cumsum(active_n_valid[:-1])])
frame_means = np.add.reduceat(dists, offsets) / active_n_valid  # (A,)
```

### Pattern 2: Precompute-then-Sweep for split-assign

**What:** Before the split-point sweep loop, precompute a `(n_groups, n_frames)` residual matrix for each (singleton, group) pair. Then the sweep iterates over split indices as cheap numpy slicing and `np.nanmean` operations.

**When to use:** `_attempt_split_assign` in `recovery.py`.

**Pseudo-code:**

```python
# BEFORE (current — expensive):
for split_idx in range(min_seg, n_frames - min_seg + 1):
    before = _slice_tracklet(singleton, 0, split_idx, ...)   # creates new Tracklet2D
    after  = _slice_tracklet(singleton, split_idx, n_frames, ...)
    for g_idx, group in enumerate(multi_groups):
        score_b = _score_singleton_against_group(before, group, ...)  # recomputes all rays
        score_a = _score_singleton_against_group(after,  group, ...)  # recomputes all rays

# AFTER (vectorized):
# Step 1: precompute per-frame residual for singleton vs each group
# residuals[g_idx, f] = per-frame residual or np.nan if frame not shared
residuals = precompute_singleton_per_frame_residuals(singleton, multi_groups, ...)
# shape: (n_groups, n_singleton_frames)

# Step 2: sweep split points over precomputed array
for split_idx in range(min_seg, n_frames - min_seg + 1):
    before_means = np.nanmean(residuals[:, :split_idx],  axis=1)  # (n_groups,)
    after_means  = np.nanmean(residuals[:, split_idx:],  axis=1)  # (n_groups,)
    # find best_before/best_after without recomputing any cast_ray
```

### Pattern 4: TimingObserver append fix

**What:** Change `write_text()` (overwrite) to `open(path, "a")` (append) so multi-chunk runs accumulate timing reports.

**Before:**
```python

```

### Anti-Patterns to Avoid

- **Scalar cast_ray inside a frame loop:** `lut.cast_ray(torch.tensor([[px, py]], dtype=torch.float32))` called for each frame is the bottleneck. Always batch all frames (or all frames × keypoints) into one call.
- **Slicing Tracklet2D inside split-point sweep:** `_slice_tracklet()` creates a new `Tracklet2D` object with sliced numpy arrays. Called for every (split_idx × group) pair, this allocates O(n_frames × n_kpts) arrays repeatedly. Precompute residuals instead.
- **Using scalar `ray_ray_closest_point` where `ray_ray_closest_point_batch` exists:** The batch version is already implemented and tested in `scoring.py`. Never call the scalar version in any loop over N > 1 pairs.

## Common Pitfalls

### Pitfall 1: reduceat requires contiguous offsets without zeros

**What goes wrong:** `np.add.reduceat(dists, offsets)` fails or gives wrong results if `offsets` contains entries where `n_valid[i] == 0` (e.g., a frame where no keypoints passed the confidence floor).

**Why it happens:** The split-assign precomputation may include frames where the singleton has no valid keypoints for a given group.

**How to avoid:** Filter out zero-count frames before building offsets, or use `np.nan` in the residual matrix for missing frames and `np.nanmean` for aggregation. The `scoring.py` pattern filters with `active_mask = n_valid > 0` before calling reduceat.

**Warning signs:** `reduceat` index out-of-bounds error, or mean residuals suspiciously close to zero.

### Pitfall 2: per_frame_confidence/consensus_centroids precomputed for wrong sub-state

**What goes wrong:** `_compute_frame_consensus` and `_compute_per_frame_confidence` in `validation.py` are called after the keep/split/evict pass. They already operate on the full `kept_tracklets` list. When vectorizing them, ensure the batching still respects the per-frame camera availability (not all cameras have every frame).

**Why it happens:** The batching template from scoring assumes two tracklets with perfectly aligned shared frames. Consensus computation involves a variable number of cameras per frame (M cameras, ~300 frames each) — the batch structure is `(cameras × frames)` not `(keypoints × frames)`.

**How to avoid:** Build the batch as "all centroids from all cameras for all frames where they appear" — a flat list of (camera_id, frame_idx, centroid_px) tuples — then group by frame using reduceat or np.unique grouping.

### Pitfall 3: Split-assign precomputation must respect overlap-check before scoring

**What goes wrong:** The precomputed residuals include scores for groups that would be blocked by the camera overlap constraint. If the sweep selects a (before_g_idx, after_g_idx) pair where one group already has a same-camera detected-frame overlap, the result is invalid.

**Why it happens:** Camera overlap depends on the segment's frame range, which changes per split_idx. In the current code, `_has_camera_overlap_with_list(before, combined)` is called per split to enforce this.

**How to avoid:** Precompute valid group membership per frame (which groups the singleton can join at each frame without camera conflict), then mask the `(n_groups, n_frames)` residual array with NaN for invalid (group, frame) combinations before the sweep.

### Pitfall 4: Timing observer append creates duplicate entries on retry

**What goes wrong:** If the pipeline crashes and is re-run, `timing.txt` grows with multiple runs interleaved. This is acceptable behavior but should be clear in the format.

**Why it happens:** Append mode is the desired behavior (accumulate per chunk), but if a run dir is reused, old chunk reports persist.

**How to avoid:** Include `run_id` and `chunk_idx` in the timing report header (the existing `report()` method already includes `run_id`). No structural change needed beyond the append fix.

### Pitfall 5: `_compute_frame_residual_keypoints` inner loop over `valid_indices`

**What goes wrong:** Even after batching the `cast_ray` call (which gives a `(N_valid, 3)` result), the current code still has an inner loop:

```python
for k in range(len(valid_indices)):
    dist, _ = ray_ray_closest_point(t_o[k], t_d[k], o_o[k], o_d[k])
    all_distances.append(dist)
```

This loop must also be replaced with `ray_ray_closest_point_batch(t_o, t_d, o_o, o_d)`.

**How to avoid:** Vectorize the entire function, including the per-keypoint inner loop.

## Open Questions

1. **Batching granularity for `_attempt_split_assign` precompute**
   - What we know: We need a `(n_groups, n_singleton_frames)` residual matrix
   - What's unclear: Whether to batch all groups at once or one group at a time; all-at-once is more complex but avoids repeated `cast_ray` calls for the singleton
   - Recommendation: Batch the singleton's rays once (for all its frames and keypoints), then reuse against each group — this avoids re-casting the singleton rays per group
